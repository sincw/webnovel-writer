# webnovel-plan Skill 分析报告

## 1. 概述

**定位**: 将总纲（总体大纲）细化为卷级骨架 + 章级大纲的规划工具。不重新设计全局故事，只做"总纲→可执行章纲"的拆解。

**触发方式**: `/webnovel-plan [卷号]`

**前置条件**: 项目根目录必须存在 `.webnovel/state.json`

## 2. 文件结构

```
.claude/skills/webnovel-plan/
├── SKILL.md                          # 主流程定义（9.3KB）
└── references/
    ├── cool-points-guide.md          # 爽点设计方法论
    ├── strand-weave-pattern.md       # 三线交织模式
    └── outlining/
        ├── outline-structure.md      # 大纲结构设计指南
        ├── chapter-planning.md       # 章节规划技巧
        ├── genre-volume-pacing.md    # 题材卷级节奏模板
        ├── plot-frameworks.md        # 剧情框架模板库
        └── conflict-design.md        # 冲突设计指南
```

## 3. 主流程（5 个 Step）

### Step 1: Load project data（加载项目数据）

**输入**:
- `.webnovel/state.json` — 项目状态（必需）
- `大纲/总纲.md` — 全书总纲（必需）
- `设定集/主角组.md`、`设定集/女主卡.md`、`设定集/反派设计.md` — 角色设定（可选）
- `.webnovel/idea_bank.json` — 创意银行/约束包（可选，v5.4.2）

**逻辑**: 如果总纲缺少卷范围/核心冲突/高潮，会要求用户先补充再继续。

### Step 2: Select volume（选择卷）

**输入**: 用户选择或指定卷号
**逻辑**:
- 从总纲提取卷名 + 章节范围供用户选择
- 确认特殊要求（基调、视角、感情线等）
- 如果总纲缺少卷名/章节范围/核心冲突/卷末高潮，先补问并更新总纲

### Step 3: Generate volume skeleton（生成卷骨架）

**参考加载（渐进式）**:
- L1（必加载）: `strand-weave-pattern.md` + `genre-profiles.md`
- L2（按需）: `cool-points-guide.md`（爽点细化时）、`reading-power-taxonomy.md`（钩子/节奏细分时）、`genre-volume-pacing.md`（电竞/直播文/克苏鲁题材时）

**三大规划维度**:

| 维度 | 内容 | 来源 |
|------|------|------|
| Strand Weave | Quest 55-65% / Fire 20-30% / Constellation 10-20% | genre-profiles + strand-weave-pattern |
| 爽点密度 | 常规章 1-2个(强度2-3) / 关键章 2-3个(强度4-5) / 高潮章 3-4个(强度6-7) | cool-points-guide |
| 约束触发 | 反套路每N章触发(N=max(5,总章数/10))、硬约束贯穿、主角缺陷≥2次、反派镜像 | idea_bank.json |

**输出**: 卷骨架 markdown，包含卷摘要、关键人物、Strand规划表、爽点密度表、伏笔规划表、约束触发规划

### Step 4: Generate chapter outlines（生成章节大纲，分批）

**分批规则**:
| 章节数 | 批次数 |
|--------|--------|
| ≤20 章 | 1 批 |
| 21-40 章 | 2 批 |
| 41-60 章 | 3 批 |
| >60 章 | 4+ 批 |

**每章生成 6 个维度**:

1. **Strand 分配** — 按卷骨架的分布，标记 Quest/Fire/Constellation
2. **爽点设计** — Quest→成就爽点、Fire→情感爽点、Constellation→认知爽点
3. **钩子设计** — 悬念钩/危机钩/承诺钩/情绪钩/选择钩/渴望钩（基于下章 Strand）
4. **反派层级** — 无/小/中/大（供 context-agent 后续使用）
5. **关键实体** — 新角色/地点/物品/势力（姓名+一句话定位）
6. **约束检查** — 是否触发反套路/硬约束/主角缺陷/反派镜像

**单章输出格式**:
```markdown
### 第 {N} 章：{标题}
- 目标: {20字以内}
- 爽点: {类型} - {30字以内}
- Strand: {Quest|Fire|Constellation}
- 反派层级: {无/小/中/大}
- 视角/主角: {主角A/主角B/女主/群像}
- 关键实体: {新增或重要出场}
- 钩子: {类型} - {30字以内}
```

**保存位置**: 每批完成后追加写入 `大纲/第{volume_id}卷-详细大纲.md`

### Step 5: Validate + save（验证 + 保存）

**5 项验证检查**:

| 检查项 | 标准 | 失败处理 |
|--------|------|----------|
| 爽点密度 | 每章≥1小爽点；每5-8章≥1关键章；每卷≥1高潮章 | 重新生成不达标批次 |
| Strand 比例 | Quest 55-65% / Fire 20-30% / Constellation 10-20%，偏差≤15% | 调整章节分配 |
| 总纲一致性 | 核心冲突贯穿、卷末高潮在最后3-5章、关键人物按计划登场 | 停止并列出冲突项 |
| 约束触发频率 | 反套路≥总章数/N次、硬约束≥50%章节、主角缺陷≥2次、反派镜像覆盖 | 补充触发点 |
| 完整性 | 每章必须有全部7个字段（目标/爽点/Strand/反派层级/视角/关键实体/钩子） | 补全缺失字段 |

**Hard Fail 条件**（必须停止）:
- 章纲文件不存在或为空
- 任一章节缺少 7 个必填字段中的任何一个
- 与总纲核心冲突或卷末高潮明显矛盾
- 约束触发频率不足（当 idea_bank 启用时）

**Rollback 策略**: 只重新生成失败批次，不覆盖整个文件。仅在 Final check 全部通过后才更新 state.json。

**状态更新**: 调用 `update_state.py --volume-planned {id} --chapters-range "{start}-{end}"`

## 4. 参考文件分析

### 4.1 strand-weave-pattern.md — 三线交织模式

**核心机制**: 将章节内容分为三条叙事线，强制交替避免节奏单调。

| 线条 | 占比 | 内容 | 硬性规则 |
|------|------|------|----------|
| Quest（主线） | 55-65% | 任务/升级/战斗/夺宝 | 不连续超过5章 |
| Fire（感情线） | 20-30% | 爱情/友情/师徒关系 | 不超过10章不出现 |
| Constellation（世界观线） | 10-20% | 设定扩展/势力/伏笔 | 不超过15章不出现 |

**状态追踪**: 通过 `state.json` 中的 `strand_tracker` 记录每条线的最后出现章节、当前主导线、切换间隔。

**提供了前30章织网模板**: 开局 Quest 密集（70-80%），第6-8章必须安排首次 Fire。

### 4.2 cool-points-guide.md — 爽点设计方法论

**六种爽点执行模式**: 装逼打脸、扮猪吃虎、越级反杀、打脸权威、反派翻车、甜蜜超预期

**核心公式 — 30/40/30 三段式**:
- 铺垫 30%: 建立预期 + 制造反差 + 信息差设置
- 兑现 40%: 触发时机 + 展现方式 + 情绪高峰
- 微反转 30%: 假结束 + 还有一手 + 余韵

**密度标准**:
| 周期 | 要求 |
|------|------|
| 每章 | ≥1 小爽点 |
| 每 5 章 | ≥1 组合爽点（2种模式叠加） |
| 每 10-15 章 | ≥1 里程碑爽点（改变主角地位） |

**打脸四步法**: 铺垫(1-2章信息差) → 挑衅(≥3次嘲讽) → 拉扯(2-3轮示弱) → 爆发(物理+精神+围观+收获)

### 4.3 genre-volume-pacing.md — 题材卷级节奏模板

仅在题材为电竞/直播文/克苏鲁时加载（L2 条件加载）。

| 题材 | Quest 占比 | Fire 占比 | Constellation 占比 | 卷结构特点 |
|------|-----------|-----------|-------------------|-----------|
| 电竞 | 50-60% | 20-30% | 15-20% | 赛程驱动，卷末3章必须有赛果兑现 |
| 直播文 | 45-55% | 20-30% | 20-30% | 反馈回路驱动，每卷≥2次可量化数据跃迁 |
| 克苏鲁 | 40-50% | 10-20% | 30-40% | 规则拼图驱动，每卷≥1次规则回收+升级 |

每个题材提供：卷结构模板（前/中/末占比）+ 卷级检查清单 + 可填写的节奏卡模板。

### 4.4 outlining/chapter-planning.md — 章节规划技巧

**章节黄金结构（3000字标准）**:
- 开头钩子 10%（300字）→ 剧情发展 50%（1500字）→ 高潮爽点 33%（1000字）→ 结尾钩子 7%（200字）

**三种章节类型及占比**:
- 爽点章 70%: 冲突→展示实力→众人震惊→收获
- 过渡章 20%: 整理收获→交代背景→布置悬念
- 刀子章 10%: 重大打击→悲痛→复仇誓言

**爽点分级标准**:
| 等级 | 频率 | 示例 |
|------|------|------|
| S级 | 每50-100章 | 打败大BOSS、飞升 |
| A级 | 每20-30章 | 打脸前未婚妻、宗门大比第一 |
| B级 | 每10章 | 秒杀挑衅者、获得神器 |
| C级 | 每3-5章 | 打脸龙套、赚灵石 |

提供了战斗章/剧情章/修炼章三种模板 + 8项自检清单。

### 4.5 outlining/outline-structure.md — 大纲结构设计指南

**三层架构**:
1. 骨架大纲（30%）— 全书起承转合，3+ 重大转折点
2. 分卷大纲（50%）— 每卷一个小目标，卷末必有高潮，卷间有过渡
3. 章节大纲（20%）— 细化到每章具体内容（可选，适合新手/卡文时）

**推荐混合流**: 骨架（全书）+ 分卷（全书）+ 章节（未来10-20章滚动补充）

**动态调整红线**: 主角人设不可突变、力量体系不可随意修改、核心主线不可轻易放弃

### 4.6 outlining/plot-frameworks.md — 剧情框架模板库

**通用框架**:
- 英雄之旅（12阶段，适用90%网文）→ 简化为三幕剧（30%/40%/30%）

**网文专属框架**:
| 框架 | 公式 | 适用场景 |
|------|------|----------|
| 打脸爽文 | 被嘲讽→隐忍→爆发→打脸→循环 | 都市/玄幻/职场 |
| 废材逆袭 | 废材→奇遇→逆袭→称霸 | 重生/系统文 |
| 扮猪吃虎 | 强者→隐藏→被小觑→爆发→震惊 | 重生/夺舍/低调流 |

**类型化框架**: 修仙类（按境界分卷）、都市异能（按能力阶段）、游戏竞技（按赛事阶段）

**四大剧情引擎**: 时间限制、空间转移、外部威胁、寻人/寻物

### 4.7 outlining/conflict-design.md — 冲突设计指南

**冲突三层次**:
1. 外部冲突（人 vs 人/势力/环境）
2. 内部冲突（人 vs 自我，道德困境/身份认同）
3. 理念冲突（价值观碰撞，正邪之争/天道之辩）

**冲突强度分级**:
| 级别 | 特征 | 触发条件 |
|------|------|----------|
| S级 | 不死不休 | 杀父之仇/灭门/世界存亡 |
| A级 | 重大利益 | 争夺传承/继承人之争 |
| B级 | 面子/地位 | 被嘲讽/排名竞争 |
| C级 | 日常摩擦 | 口角/误会 |

**升级公式**: 小冲突 → 误会加深 → 利益纠葛 → 不死不休

**密度标准**: 每10章≥1个B级冲突，每5章1个小冲突+每20章1个大冲突

## 5. 数据流

```
输入层
─────────────────────────────────────────────────
.webnovel/state.json          ← 项目状态
大纲/总纲.md                   ← 全书总纲
设定集/*.md                    ← 角色设定（可选）
.webnovel/idea_bank.json       ← 创意约束（可选）
.claude/references/genre-profiles.md  ← 题材配置

处理层（Step 1-4）
─────────────────────────────────────────────────
Step 1: 加载数据 + 校验完整性
    ↓
Step 2: 用户选择卷 + 确认范围
    ↓
Step 3: 生成卷骨架
         ├─ Strand Weave 分配
         ├─ 爽点密度规划
         └─ 约束触发规划
    ↓
Step 4: 分批生成章节大纲
         ├─ 每章 7 字段
         └─ 每批写入文件

验证层（Step 5）
─────────────────────────────────────────────────
5 项检查 → 全部通过 → 更新 state.json
              ↓ 失败
         Rollback 失败批次 → 重新生成

输出层
─────────────────────────────────────────────────
大纲/第{N}卷-详细大纲.md       ← 卷骨架 + 章节大纲
.webnovel/state.json           ← 更新 volume_planned
```

## 6. 参考加载策略（渐进式披露）

Skill 采用三级加载策略，避免一次性加载全部参考文件导致上下文膨胀：

| 级别 | 时机 | 加载内容 | 条件 |
|------|------|----------|------|
| L0 | Step 1-2 | 无参考文件 | 卷/范围未确认前不加载 |
| L1 | Step 3 前 | strand-weave-pattern.md + genre-profiles.md | 必加载 |
| L2 | Step 3 按需 | cool-points-guide.md | 爽点结构需要细化时 |
| L2 | Step 4 按需 | reading-power-taxonomy.md | 钩子/节奏需要细分时 |
| L2 | Step 3/4 按需 | genre-volume-pacing.md | 仅电竞/直播文/克苏鲁题材 |

## 7. 与上下游 Skill 的衔接

### 上游：webnovel-init
- 提供 `state.json`、`总纲.md`、设定集、`idea_bank.json`
- plan 依赖 init 产出的总纲卷结构和创意约束

### 下游：webnovel-write
- 读取 `大纲/第{N}卷-详细大纲.md` 中的章节大纲
- context-agent 使用章节的 7 字段（特别是反派层级、钩子）生成创作任务书
- 章节大纲中的"钩子"字段是规划意图，实际写作后由 data-agent 记录到 `chapter_meta[N].hook`

## 8. 关键设计特点总结

1. **渐进式加载** — 不一次性灌入所有参考，按 Step 需要分级加载（L0/L1/L2），控制上下文消耗
2. **分批生成** — 章节大纲按批次生成并逐批保存，降低单次生成失败的影响范围
3. **五重验证** — 爽点密度/Strand比例/总纲一致性/约束触发/完整性，全部通过才更新状态
4. **安全回滚** — Hard Fail 时只重写失败批次，不覆盖已通过的内容
5. **约束继承** — 从 idea_bank.json 继承创意约束，计算触发频率并分配到具体章节
6. **结构化输出** — 每章固定 7 字段格式，为下游 context-agent 和 webnovel-write 提供标准化输入
